# Mutation Feedback and Idempotency Design

Date: 2026-07-12
Status: approved for implementation

## Problem

Admin mutations currently provide inconsistent or invisible feedback. A user can click repeatedly while a server action is pending, creating duplicate FAQ records and duplicate sync work. Mutations also need to communicate when the saved change becomes available to the live chatbot.

## Goals

- Give every create, update, delete, invitation, role, and tab-access mutation a consistent centered modal.
- Keep success/error modals open until the user explicitly clicks `Done` or `Close`.
- Show a loading animation while persistence and chatbot synchronization are pending.
- Confirm `Live chatbot updated` only after the sync job succeeds.
- Make duplicate submissions idempotent and treat exact duplicate creates as updates.
- Make destructive actions use a stronger confirmation step.
- Preserve server-side authorization and provide structured mutation results to the UI.

## Non-goals

- Replacing Clerk or Dify.
- Making the browser wait forever for an unavailable chatbot provider.
- Removing audit logging or existing revalidation behavior.

## Mutation inventory

The shared lifecycle covers:

- Faculties, coaches, programmes, course modules, resources/links, document resources, and FAQs.
- User invitations, user role updates, and role/tab-access updates.
- Deletes for faculties, coaches, programmes, course modules, resources, and FAQs.

## UX design

All mutation surfaces use a shared modal primitive:

1. Centered dialog, blurred backdrop, focus trap, Escape handling, and accessible status text.
2. `Submitting` state disables the form and submit control immediately.
3. `Saved` state confirms database persistence and displays the affected record.
4. `Syncing` state shows a spinner and `Updating the live chatbot...`.
5. `Complete` state displays `Live chatbot updated` and an explicit `Done` button.
6. `Error` state displays a safe error message, preserves the form context, and offers `Try again` or `Close`.

Delete uses a separate confirmation variant with explicit record name, consequence text, and `Cancel`/`Delete permanently` controls before entering the same progress lifecycle.

Existing create/edit dialogs are migrated to the shared centered primitive; no mutation uses an inline-only success message.

## Persistence and idempotency

Every mutation receives a request id generated by the client and validated server-side. Repeated requests with the same request id return the original result and do not repeat writes or enqueue duplicate jobs.

Exact duplicate creates use domain identity keys and become updates:

- FAQ: normalized faculty, category, and question identity; answer/content is updated.
- Other seeded entities: existing `seedKey` or domain-unique key.
- Resources: existing seed key, with document metadata included in the update path.

Database unique constraints remain the final backstop. Upsert/update behavior must preserve authorization, audit records, cache invalidation, and sync enqueueing exactly once.

## Sync contract

Mutation responses return a structured result containing mutation id, record id, persistence status, and sync job id. The client observes job status through the existing sync status path (polling first; realtime transport is not required for this phase). A successful save without a successful sync never displays `Live chatbot updated`.

If sync fails, the modal remains open with a retry action and a clear `Saved, chatbot update pending/failed` message. Retry reuses the record and creates no duplicate record.

## Implementation boundaries

- Shared client mutation controller owns state, disabled controls, modal transitions, and status polling.
- Shared server mutation result type standardizes action responses.
- Server action helpers own request-id deduplication, domain upserts, audit behavior, cache invalidation, and sync enqueueing.
- Existing domain actions remain authorization boundaries; the shared layer must not bypass them.
- Add or update database migrations for request-id deduplication records and missing domain uniqueness constraints where needed.

## Verification

- Unit tests for modal state transitions, duplicate request handling, normalized FAQ identity, and retry behavior.
- Server-action tests proving the same request id writes once and enqueues one sync job.
- Route-level tests for every mutation surface, including double-click/Enter submission and delete cancellation.
- Browser smoke tests proving the modal is centered, background-blurred, focusable, and remains open until `Done`.
- End-to-end FAQ scenario: submit once, repeat the same request, verify one row, one sync job, visible progress, and final chatbot-updated confirmation.
